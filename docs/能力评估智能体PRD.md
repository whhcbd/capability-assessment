# 能力评估智能体 PRD

> 当前状态：本文档是能力评估智能体首版 PRD，部分内容仍记录“mock/RAG placeholder”和固定问答阶段目标。当前实现已迁移到 `app/`，主流程为简历/心仪职业、48 题自评问卷、真实 Ability API、真实 RAG API 和能力雷达报告。本文保留作为产品边界和能力 schema 的历史依据，当前运行说明以 `README.md` 和 `ARCHITECTURE.md` 为准。

## 1. 背景与目标

项目当前从第一阶段的就业服务信息系统，继续推进到面向 C 端学生的智能化求职系统。四个智能体分别覆盖职位推荐、能力评估、简历助手和面试陪练。

能力评估智能体的定位是四个智能体中的能力模型底座：它不直接爬取职位、不做岗位推荐，也不负责输出训练方案，而是通过多种评估方式生成学生个人职业能力画像，并预留通过 RAG 生成职业/岗位能力需求图的能力，让其他智能体可以基于同一套能力数据继续工作。

首版目标：

- 面向本科生和研究生，帮助学生理解自己的职业能力结构。
- 通过简历/经历文本、对话式问答、自评问卷三类输入生成能力雷达图。
- 输出可解释的能力分数、证据摘要和置信度。
- 首版通过 mock/RAG placeholder 生成 `role_capability_profile`，后续再接入真实 RAG。
- 产出标准化 `capability_profile` 和 `role_capability_profile`，供职位推荐、简历助手、面试陪练读取。
- 为后续独立智能体 demo 提供清晰的产品流程和接口约定。

## 2. 用户与场景

### 2.1 目标用户

- 本科生：准备实习、校招或第一次职业选择的学生。
- 研究生：准备转专业方向、校招、实习或职业转型的学生。

首版演示默认用户可以设定为：

```text
Demo User：本科/研究生在读学生，目标方向偏互联网公司，但尚未锁定具体岗位。
```

### 2.2 核心场景

学生希望知道：

- 我的职业能力结构是什么样的？
- 我在哪些能力上表现更强？
- 系统为什么给出这些分数？
- 哪些能力评估结果更可信，哪些还需要补充信息？

能力评估智能体输出用户能力图和岗位能力需求图后，其他智能体可以继续使用：

- 职位推荐智能体：负责从互联网或其他渠道抓取职位机会，读取能力评估智能体生成的岗位能力需求图和用户能力画像，再做推荐排序和解释。
- 简历助手智能体：读取能力画像，判断简历中哪些能力证据不足。
- 面试陪练智能体：读取能力画像，选择更适合的面试训练重点。

## 3. 模块边界

### 3.1 本模块负责

- 定义首版职业能力维度体系。
- 收集并分析学生个人经历、问答回答和自评量表。
- 生成用户能力雷达图所需的分数数据。
- 输出每项能力的证据来源、证据摘要和置信度。
- 输出标准化 `capability_profile` JSON。
- 首版通过 mock/RAG placeholder 生成职业/岗位能力需求图，并预留真实 RAG 接入点。
- 输出标准化 `role_capability_profile` JSON。

### 3.2 本模块不负责

- 不负责从互联网爬取职位列表。
- 不负责岗位推荐结果。
- 不负责岗位匹配度报告。
- 不负责“你适合哪些岗位”的推荐解释。
- 不负责“距离目标岗位还差什么”的训练建议。
- 不负责 PDF/Word 简历上传解析。
- 不负责完整登录、用户权限和信息系统集成。

### 3.3 与岗位能力需求图的关系

岗位能力需求图由能力评估智能体负责生成。首版 Demo 不实现真实 RAG，只使用 mock/RAG placeholder 展示能力需求图和接入位置；后续再通过 GitHub 可用 RAG 仓库或自研 RAG 管线接入真实检索。职位推荐智能体负责从互联网或其他渠道获取职位机会，并把这些职位机会交给能力评估智能体或岗位能力模型能力来抽取岗位要求。

这样做的原因是：用户能力图和岗位能力需求图必须使用同一种能力语言，后续才能比较。例如：

```text
用户能力图：logical_analysis = 72
岗位需求图：logical_analysis.required_level = 80
```

职位推荐智能体可以据此判断用户在该能力上与目标岗位存在 8 分差距。能力评估智能体提供两类基础数据：用户能力画像和岗位能力需求图；职位推荐智能体负责用这些数据完成推荐排序、匹配解释和推荐结果展示。

## 4. 首版能力维度体系

首版定义 8 个一级能力维度。每个维度使用稳定英文 key，中文名用于产品展示。

| Key | 中文名 | 定义 |
| --- | --- | --- |
| `communication_expression` | 沟通表达能力 | 清晰表达观点、组织语言、传递信息、回应问题的能力。 |
| `logical_analysis` | 逻辑分析能力 | 拆解问题、建立结构、识别因果关系和形成判断的能力。 |
| `learning_adaptability` | 学习适应能力 | 面对新任务、新环境、新工具时快速学习和调整的能力。 |
| `execution_ownership` | 执行推进能力 | 将目标拆成行动、持续推进、按时交付并承担结果的能力。 |
| `collaboration_leadership` | 协作与领导力 | 与他人协同、推动团队、协调资源和处理分歧的能力。 |
| `self_awareness_motivation` | 自我认知与职业动机 | 理解自身兴趣、价值观、优势短板和职业选择动机的能力。 |
| `data_digital_literacy` | 数据与数字化思维 | 使用数据、工具和数字化方法理解问题、支持决策的能力。 |
| `business_industry_understanding` | 商业与行业理解 | 理解行业、商业模式、用户需求和组织目标的能力。 |

分数范围：

- `0-39`：证据不足或能力表现较弱。
- `40-59`：具备基础表现，但稳定性和深度不足。
- `60-79`：具备较明显能力表现，可用于求职材料和面试表达。
- `80-100`：能力表现突出，有较强证据支撑。

置信度范围：

- `0.00-0.39`：证据较少，需要补充评估。
- `0.40-0.69`：有一定证据，但仍可能偏主观。
- `0.70-1.00`：多来源证据一致，评估可信度较高。

## 5. 评估方式

### 5.1 简历/经历文本分析

首版不做文件上传。用户在文本框中粘贴：

- 简历正文。
- 实习经历。
- 项目经历。
- 社团/竞赛/课程经历。
- 自我介绍。

系统分析目标：

- 抽取经历片段。
- 识别经历中体现的能力维度。
- 生成初步能力证据。
- 标记证据强弱。

示例：

```text
用户描述：作为课程项目负责人，协调 4 名同学完成用户调研和原型设计。
可能证据：collaboration_leadership、execution_ownership、communication_expression
```

### 5.2 对话式能力问答

首版采用固定问题 + LLM rubric 评分，不做完全自由对话。

建议固定 6 个问题：

1. 请介绍一个你主导或深度参与的项目，并说明你具体负责什么。
2. 遇到一个复杂问题时，你通常如何拆解和推进？
3. 请描述一次你需要和他人协作但意见不一致的经历。
4. 请讲一个你快速学习新领域、新工具或新方法的例子。
5. 你目前选择互联网方向的主要原因是什么？
6. 请举例说明你如何使用数据或事实支持自己的判断。

LLM 输出要求：

- 每题只评估相关能力维度。
- 输出结构化 JSON。
- 每项评分必须附简短证据。
- 不输出岗位推荐和训练建议。

### 5.3 自评问卷/量表

自评问卷用于快速补充学生的主观倾向和自我认知。首版不引入复杂心理测评版权内容，只使用自研职业能力自评题。

建议形式：

- 每个能力维度 2-3 道题。
- 使用 1-5 分 Likert scale。
- 自评分不直接等同最终能力分，只作为辅助证据。

示例题：

```text
我能在 2 分钟内清楚说明一个复杂项目的目标、过程和结果。
1 非常不同意 / 2 不同意 / 3 一般 / 4 同意 / 5 非常同意
```

### 5.4 岗位能力需求图 RAG 占位

岗位能力需求图用于描述某个职业方向或具体岗位需要哪些能力、每项能力的重要程度和要求水平。

后续真实 RAG 输入来源：

- 具体 JD 文本。
- 互联网公开岗位描述。
- 行业岗位介绍材料。
- 职业能力模型资料。
- 学校或顾问沉淀的岗位经验文档。

首版 Demo 生成目标：

- 将岗位要求映射到统一 8 个能力维度。
- 为每项相关能力生成 `required_level`。
- 为每项相关能力生成 `weight`。
- 输出岗位能力要求摘要和资料依据。
- 不做岗位推荐排序。
- 不实现真实检索，使用内置示例或 mock JSON 作为 RAG 占位输出。

首版页面需要明确展示：

```text
当前为 RAG 占位演示：已预留 JD 输入、资料检索和岗位能力抽取位置，真实 RAG 后续接入。
```

示例：

```text
输入：互联网产品经理实习生 JD
输出：communication_expression、logical_analysis、data_digital_literacy 等能力要求及权重
```

## 6. 输出结果

### 6.1 学生可见输出

首版学生端展示：

- 个人能力雷达图。
- 8 个能力维度分数。
- 每项能力的简短解释。
- 每项能力的证据摘要。
- 每项能力的置信度。
- 信息不足提示，例如“该能力目前主要来自自评，建议补充项目经历或问答评估”。

不展示：

- 岗位推荐列表。
- 岗位匹配度报告。
- 训练路径。
- 课程推荐。

### 6.2 标准化能力画像 JSON

能力评估智能体输出 `capability_profile` 和 `role_capability_profile`，供其他智能体调用。

```json
{
  "user_id": "demo_user_001",
  "profile_version": "v1",
  "generated_at": "2026-06-05T00:00:00+08:00",
  "capability_profile": {
    "communication_expression": {
      "score": 78,
      "confidence": 0.72,
      "evidence_sources": ["resume_text", "dialogue_answer", "self_assessment"],
      "evidence_summary": "用户有社团展示和项目沟通经历，回答结构较清晰。"
    },
    "logical_analysis": {
      "score": 70,
      "confidence": 0.66,
      "evidence_sources": ["dialogue_answer", "resume_text"],
      "evidence_summary": "用户能描述问题拆解过程，但量化分析证据较少。"
    }
  }
}
```

岗位能力需求图示例：

```json
{
  "role_id": "internet_product_intern",
  "role_name": "互联网产品经理实习生",
  "profile_version": "v1",
  "source_type": "mock_rag_placeholder",
  "rag_status": "placeholder",
  "source_refs": ["demo_jd_text", "mock_industry_knowledge"],
  "requirements": {
    "communication_expression": {
      "required_level": 75,
      "weight": 0.2,
      "requirement_summary": "需要清晰表达产品思路、用户问题和方案取舍。"
    },
    "logical_analysis": {
      "required_level": 80,
      "weight": 0.25,
      "requirement_summary": "需要拆解业务问题、用户路径和产品指标。"
    }
  }
}
```

### 6.3 LLM 评分输出约束

LLM 对单次问答或经历文本分析时，应输出可合并的局部结果。

```json
{
  "source_type": "dialogue_answer",
  "source_id": "question_01",
  "capability_evidence": [
    {
      "capability_key": "communication_expression",
      "score": 76,
      "confidence": 0.68,
      "evidence_summary": "用户能按背景、行动、结果描述项目经历。"
    }
  ]
}
```

## 7. 产品流程

首版能力评估流程：

```text
进入能力评估智能体
-> 查看评估说明
-> 粘贴简历/经历文本
-> 系统生成初步能力证据
-> 回答固定能力评估问题
-> 完成自评问卷
-> 系统合并三类证据
-> 生成能力雷达图
-> 输入或选择岗位/JD
-> RAG placeholder 生成岗位能力需求图
-> 展示能力明细、证据摘要和置信度
-> 导出 capability_profile 与 role_capability_profile JSON
```

首版 demo 页面建议包含 4 个区域：

1. 输入区：简历/经历文本、自评问卷、固定问答。
2. 分析状态区：显示当前已完成的评估来源。
3. 雷达图区：展示 8 个能力维度分数。
4. 岗位能力区：输入 JD 或选择示例岗位，展示岗位能力需求图。
5. 结构化结果区：展示 JSON 或可复制的能力画像。

## 8. 与其他智能体协作

### 8.1 职位推荐智能体

读取：

- `capability_profile`
- `role_capability_profile`
- 统一能力维度 key

用途：

- 从互联网爬取或整理职位机会。
- 调用能力评估智能体生成或读取岗位能力需求图。
- 比较用户能力图与岗位能力需求图。
- 生成岗位推荐和匹配解释。

边界：

- 职位推荐智能体负责职位抓取、推荐排序和匹配结论展示。
- 能力评估智能体负责用户能力图和岗位能力需求图。

### 8.2 简历助手智能体

读取：

- `capability_profile`
- 每项能力的 `evidence_summary`

用途：

- 判断简历中哪些能力已有证据。
- 判断哪些能力缺少可写经历。
- 帮助用户挖掘经历素材。

边界：

- 简历助手负责简历内容组织和表达优化。
- 能力评估智能体不生成简历。

### 8.3 面试陪练智能体

读取：

- `capability_profile`
- 低分或低置信度能力项

用途：

- 配置面试训练重点。
- 面试结束后可把表现结果回写为新的能力证据。

边界：

- 面试陪练负责提问、追问和面试反馈。
- 能力评估智能体负责接收并合并面试表现证据。

## 9. 独立 demo 要求

首版 demo 推荐采用独立页面，不接入现有 `career` 前端和后端。

默认设置：

- 默认用户：`demo_user_001`
- 无登录系统。
- 不做 PDF/Word 上传。
- 简历输入使用文本粘贴。
- 对话问答使用固定问题。
- LLM 使用固定 rubric 输出 JSON。
- RAG 使用 mock placeholder，不实现真实检索。
- 雷达图展示学生本人视角。

演示重点：

- 能力评估流程完整。
- 岗位能力需求图可以通过 JD 输入和 RAG placeholder 展示。
- 输出结果可解释。
- 能力画像可被其他智能体读取。
- 不是一次性玩具 demo，而是未来可接入信息系统的数据底座。

## 10. 验收标准

PRD 验收：

- 明确说明能力评估智能体的模块边界。
- 明确说明它是四个智能体的能力模型底座。
- 明确说明首版输出用户能力画像和岗位能力需求图，不输出推荐和训练建议。
- 明确给出 8 个能力维度和稳定英文 key。
- 明确给出 `capability_profile` JSON 结构。
- 明确给出 `role_capability_profile` JSON 结构。
- 明确给出从输入到雷达图输出的演示流程。
- 明确说明首版 RAG 是占位，不实现真实检索。

后续 demo 验收：

- 输入简历/经历文本后，能生成初步能力证据。
- 完成固定问答后，能更新能力评分。
- 完成自评问卷后，能生成完整雷达图。
- 输入或选择 JD 后，能展示 RAG 占位区和 mock 岗位能力需求图。
- 能展示能力明细、证据摘要和置信度。
- 能导出或展示 `capability_profile` 与 `role_capability_profile` JSON。
- 不依赖完整登录系统即可演示。
