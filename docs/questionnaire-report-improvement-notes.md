# 问卷与个人界面优化问题归纳

> 来源：2026-06-14 会议反馈。本文先归纳问题与期望效果，作为后续实现与验收依据。

## 背景

当前主流程为：

```text
简历与理想岗位
-> 是否填写问卷
-> 个人界面
```

本轮反馈主要集中在两个区域：

- 题目测试问卷：答题导航、漏题处理、快速演示模式。
- 个人界面：雷达图对比、能力明细表达、可执行行动建议。

## QA-01：修复问卷快速点击跳题问题

**问题描述**

在题目测试问卷中，用户快速点击选项时，会直接跳过中间题目。提交时系统再跳回之前漏答的题目，用户补答后还需要点击很多次“下一题”才能重新到达提交位置。

**影响**

- 用户以为自己已经顺利答完，实际存在漏题。
- 演示时容易暴露流程不稳定。
- 当前“必须在最后一题提交”的交互放大了漏题后的操作成本。

**期望行为**

- 快速点击选项时不能跳过未稳定记录的题目。
- 每次作答只更新当前题答案和答题状态。
- 自动进入下一题时，需要避免重复触发或并发触发。
- 如果存在漏题，用户可以通过题号导航直接跳到漏题，不需要从当前位置连续点“下一题”。

**验收标准**

- [ ] 连续快速点击同一题选项，不会导致题号连续跳多题。
- [ ] 连续快速切换不同选项，最终只保留该题最后一次选择。
- [ ] 存在漏题时，提交按钮给出明确提示，并能定位到第一个漏题。
- [ ] 补完漏题后，可以直接提交，不需要重新走到最后一题。

## QA-02：增加可收起右侧题号导航栏

**目标**

参考会议截图，在问卷页右侧增加一个可展开/收起的答题导航栏，用题号网格展示问卷完成状态。

**交互要求**

- 右栏支持打开和收起。
- 右栏展示全部题号。
- 已完成题目变色，并显示完成标记。
- 当前题高亮。
- 未完成题保持未答状态。
- 点击题号可以跳转到对应题目。
- 右栏放置问卷提交入口。

**提交规则**

- 下一题不一定要求当前题已作答，用户可以跳题浏览。
- 问卷提交不要求用户停留在最后一题。
- 问卷提交必须要求所有题目完成。
- 如果未全部完成，提交时显示剩余未答数量，并定位到第一个未答题。

**响应式要求**

- 桌面端：右栏固定在问卷内容右侧，可收起。
- 移动端：右栏建议作为抽屉或底部弹层出现，避免挤压题目正文。

**验收标准**

- [ ] 48 题模式下，右栏可以完整查看所有题号。
- [ ] 10 题快速模式下，右栏只展示 10 个题号。
- [ ] 已答题、当前题、未答题三个状态视觉上可区分。
- [ ] 提交按钮在任意题目位置都可用，但未完成时不能提交。

## QA-03：增加 10 题快速作答模式

**问题描述**

当前问卷固定为 48 题，测试和演示成本较高。需要提供一个更短的快速模式，用于快速跑通“简历 + 岗位 JD + 问卷 + 个人界面”的完整链路。

**期望流程**

用户填写个人简历并选择行业 JD 后，进入问卷调查时先选择：

- 快速模式：10 题。
- 详细模式：48 题。

两种模式只有题量不同，其他流程保持一致。

**产品约束**

- 快速模式主要服务测试和演示。
- 快速模式仍然需要输出完整个人界面。
- 快速模式的结果需要在界面上标注题量或评估模式，避免和 48 题详细评估混淆。
- 不引入新的 mock 主流程；仍然使用当前正式问卷与评估链路。

**验收标准**

- [ ] 简历和岗位 JD 完成后，问卷页可选择快速模式或详细模式。
- [ ] 快速模式只展示 10 题。
- [ ] 详细模式保留 48 题。
- [ ] 两种模式都能完成提交并进入个人界面。
- [ ] 个人界面能识别本次问卷模式和题量。

## RP-01：合并个人与职业雷达图

**问题描述**

当前个人能力雷达图和职业能力雷达图分开展示，对比差距不够直观。

**期望行为**

个人界面中的雷达图改为同一张图展示，并提供显示选项：

- 只显示个人能力图。
- 只显示职业要求图。
- 同时显示个人能力与职业要求，叠加对比。

**默认建议**

默认显示“两者叠加”，因为这是最直接体现差距的视图。

**验收标准**

- [ ] 雷达图可以在个人、职业、两者叠加之间切换。
- [ ] 两条图形样式有明确区分，例如颜色、线型、透明度或图例。
- [ ] 切换显示不会改变原始评分数据。
- [ ] 叠加视图能清楚看出每个能力维度的差距。

## RP-02：雷达图内显示具体数字

**问题描述**

当前雷达图主要展示形状，用户需要额外查看明细才能知道具体分数。

**期望行为**

在雷达图中直接标注具体数字：

- 个人能力分数。
- 职业要求分数。
- 在叠加视图下，两类数字需要能区分。

**设计注意**

- 数字不能遮挡能力维度标签。
- 移动端不能因为数字过多导致重叠严重。
- 如果空间不足，可以优先显示当前选中图层的分数，并通过悬浮或点击展示另一组分数。

**验收标准**

- [ ] 单图模式下，每个能力维度能看到对应分数。
- [ ] 叠加模式下，用户能区分个人分数和职业分数。
- [ ] 数字在桌面和移动端不明显重叠。

## RP-03：能力明细改为 3 大展示板块

**问题描述**

当前能力明细类似表格，信息密度较高，但不利于面试准备和后续行动。

**期望结构**

每个能力维度的明细改为 3 大展示板块：

1. 岗位应用场景
   - 说明该能力在对应工作里的作用。
   - 给出实际应用实例。
   - 帮助用户准备面试表达。

2. 个人能力评估
   - 结合用户简历和自我打分内容。
   - 对该能力做水平判定。
   - 说明主要证据、优势和不足。

3. 针对性改进建议
   - 根据个人能力与岗位要求差距给出优化方向。
   - 建议要具体到可学习、可训练、可补充经历的动作。

**展示建议**

- 桌面端可以做成三列或三块并排展示。
- 移动端按“岗位应用场景 -> 个人能力评估 -> 针对性改进建议”纵向排列。
- 每个能力维度保留分数、差距、置信度等关键指标，但不再以表格作为主要表达。

**验收标准**

- [ ] 每个能力维度都有上述 3 个板块。
- [ ] 板块内容围绕用户选择的目标岗位生成，不是通用空话。
- [ ] 个人能力评估能引用简历、自评或评估证据。
- [ ] 改进建议至少包含 1 条可执行动作。

## RP-04：生成汇总行动清单

**目标**

在个人界面中汇总全部能力的优化建议，生成一份“马上能做”的行动清单，清晰列出后续需要学习、训练、补充的内容。

**内容要求**

行动清单需要具备可执行性：

- 明确做什么。
- 明确为什么做，对应哪个能力差距。
- 明确建议优先级。
- 明确可以马上开始的第一步。

**建议结构**

```text
优先级
-> 行动项
-> 对应能力
-> 目标岗位场景
-> 今天可以开始的第一步
```

**示例表达方向**

- 学习类：补齐某项岗位要求中的基础知识。
- 训练类：用小项目、案例分析、模拟面试训练某项能力。
- 经历补充类：在简历中补充能证明能力的项目结果、指标和协作过程。
- 面试准备类：把已有经历整理成 STAR 表达。

**验收标准**

- [ ] 个人界面包含汇总行动清单模块。
- [ ] 行动清单覆盖主要能力差距，不只是逐项重复能力明细。
- [ ] 每条行动项都能马上执行。
- [ ] 行动项能追溯到具体能力维度或岗位要求。

## RP-05：个人界面问卷模式弹窗

**问题描述**

个人界面中进入问卷作答时，不应默认进入 48 题详细模式。用户可能只是想快速补测个人雷达，也可能想完成完整问卷。

**期望行为**

- 点击个人界面中的问卷入口时，先弹窗选择模式。
- 弹窗只包含三个操作：快速模式 10 题、详细模式 48 题、取消。
- 如果已经有个人雷达，重新作答后覆盖当前个人能力画像，不做历史记录。

**验收标准**

- [ ] 个人界面问卷入口不会直接进入 48 题。
- [ ] 用户可以在弹窗中选择快速模式或详细模式。
- [ ] 取消弹窗后仍停留在个人界面。
- [ ] 重新作答提交后更新当前个人雷达。

## RP-06：个人界面二次重构

**问题描述**

当前“相对优势”和“优先补齐”两个模块不符合新的个人界面规划，需要删除。个人界面应更聚焦雷达对比和完整能力明细。

**期望结构**

```text
岗位信息 + 能力雷达
-> 全部展开的能力明细
-> 可执行行动清单
```

**验收标准**

- [ ] 不再展示“相对优势”。
- [ ] 不再展示“优先补齐”。
- [ ] 8 个能力明细全部展开。
- [ ] 行动清单放在能力明细后方。

## RP-07：雷达图标签旁显示分数

**问题描述**

雷达图内部点位旁显示数字会显得拥挤，视觉效果不好。分数应靠近“逻辑分析”等能力标签，用户看标签时即可看到对应数值。

**期望行为**

- 雷达内部不再显示分数圆点。
- 能力标签旁显示分数。
- 叠加模式显示 `个人分/职业分`，例如 `逻辑分析 72/85`。
- 只看个人模式只显示个人分。
- 只看职业模式只显示职业分。

**验收标准**

- [ ] 雷达内部没有分数圆点。
- [ ] 能力标签旁显示正确分数。
- [ ] 叠加模式能看出个人分和职业分。

## 待定事项

### TBD-01：职业个性化 AI 问卷

后续希望探索“职业个性化 AI 问卷”：系统根据用户选择的目标岗位、JD 和岗位能力需求，动态生成更贴近岗位的问卷题目。

当前只记录为待定事项，本轮不实现、不设计接口。

后续需要确认：

- 个性化 AI 问卷是替代固定 10/48 题，还是作为第三种问卷模式。
- 题目生成是否依赖 RAG 的岗位能力需求和检索资料。
- 动态生成题目是否需要人工审核、题目缓存、版本记录和复测一致性。
- 个性化问卷提交后，如何与现有 `capability-evidence` 评分链路兼容。

### TBD-02：RAG 内容填充

当前 RAG 的明确使用范围是职业雷达：前端调用 `POST /assessments/role-profile`，后端通过 RAG / DeepSeek 生成岗位能力需求图。

个人雷达目前不是 RAG 生成：前端调用 `POST /assessments/capability-evidence`，后端基于已保存简历和问卷答案生成个人能力证据，并合并为个人能力画像。

如果后续要做 RAG 内容填充，可以考虑：

- 在能力明细的“岗位应用场景”中引入 RAG 检索到的岗位资料。
- 让 RAG 输出岗位应用实例、面试场景、来源依据等结构化字段。
- 在前端展示 `source_refs` 或 `retrieved_chunks`，帮助说明职业雷达和岗位场景的来源。

当前本轮不扩展后端 API，不改变岗位画像 schema。

## 后续拆分建议

建议按以下顺序实现：

1. 先修 QA-01，解决快速点击跳题和漏题提交路径问题。
2. 再做 QA-02，把右侧题号导航和任意位置提交建立起来。
3. 接着做 QA-03，降低测试和演示成本。
4. 然后做 RP-01 和 RP-02，提升个人界面差距对比效率。
5. 最后做 RP-03 和 RP-04，把报告内容从“展示结果”升级为“帮助用户准备面试和改进”。
